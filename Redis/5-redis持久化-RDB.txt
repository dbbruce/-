1、RDB（redis database）
    指定的时间间隔内将内存中的数据集快照写入磁盘
    也就是行话讲的snapshot快照，它恢复是将快照文件直接写入内存

    redis单独创建一个子进程进行持久话，会先将数据写入到一个临时文件中，待持久化过程结束了，在用临时文件替换上次持久化好的文件
    整个过程中，主进程不进行任何IO操作，确保极高的性能
    如果需要大规模数据恢复，且对于数据恢复的完整性不是非常敏感，那RDB方式比AOF方式高效，RDB的缺点是最后一次持久化后的数据可能丢失；

2、Fork
    fork的作用是复制一个与当前进程一样的进程，新进程的所有数据（变量，环境变量，程序计数器）数值都和原进程一致；
    但是是一个全新进程，作为原进程的子进程

3、RDB保存的是dump.rdb文件，redis.conf里配置；

4、lsof -i :6379

5、关闭redis
    shutdown
6、save/bgsave
save时只管报错，其他不管，全部阻塞
bgsave后台异步进行快照操作，同时可以响应客户端请求，通过lastsave命令获取最后一次成功执行快照时间；

7、flushall 也会产生dump.rdb文件，但是文件内容为空，没有意义；

8、优势，适合大规模数据恢复，对数据一致性要求不高；

9、劣势，最后一次的快照会丢失；

10、停止
动态停止RDB保存规则的方法：
redis-cli config set save ""
